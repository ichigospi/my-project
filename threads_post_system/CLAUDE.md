# Threadsポスト作成システム — 全体方針

Threads(スレッズ)の **投稿作成 / 競合キャッチアップ / 競合分析 / 自アカ閲覧数分析** を行うAIチームシステム。

## 大原則:ユーザーは統括にのみ指示を出す

ユーザー(あなた)からの指示・ナレッジ提供は **すべて統括(ディレクター)が受け取る**。
AIはユーザーから指示を受けたら、必ず以下の順で動く:

1. `prompts/director.md` を読み、統括として指示を解釈・ルーティングする
2. 該当する担当ロールのプロンプト(`prompts/*.md`)を読み、そのロールとして実行する
3. 成果物を統括として取りまとめ、ユーザーに提案・報告する
4. **必ず応答の最後に【次のアクション】を付け、次に必要なものを自分から要求する**
   (指示待ち禁止。優先順位ロジックは `prompts/director.md` 参照)

ユーザーが個別ロールを名指ししなくても、統括のルーティング表で自動的に組織が回る。

## トリガー

ユーザーから以下の指示があった場合、本システムを使用する:

- 「スレッズの投稿を作って」「Threadsのポストを作って」
- 「（競合の）伸びてる投稿をキャッチアップして / 分析して」
- 「自アカの閲覧数を分析して」「なんで伸びなかったか見て」
- 「参考ポストに追加して」「教材を読み込ませて」「自アカ情報を更新して」

## 組織図(6ロール)

| ロール | プロンプト | 責務 |
|---|---|---|
| ①統括(ディレクター) | `prompts/director.md` | 唯一の窓口。指示・ナレッジの受付、タスク振り分け、提案・報告、最終判断 |
| ②リサーチャー | `prompts/researcher.md` | 競合・トレンド投稿の収集(解釈しない) |
| ③競合アナリスト | `prompts/trend_analyst.md` | 競合投稿の構造分解・伸び要因分析・転用案 |
| ④自アカアナリスト | `prompts/performance_analyst.md` | 自アカ閲覧数から伸びた/伸びなかった要因を分析 |
| ⑤ライター | `prompts/writer.md` | 投稿ドラフト作成(自己判定しない) |
| ⑥レビュアー | `prompts/reviewer.md` | 独立チェック・差し戻し権限 |

ナレッジ取り込み(教材PDF・URL・コピペの分類整理)は **統括が兼任**する
(手順は `prompts/director.md` 内の「ナレッジ受付フロー」参照)。

### ③と④を分ける理由

- ③競合分析は「収集データが溜まったとき」に走る(入力: 他人の投稿)
- ④自アカ分析は「投稿後の閲覧数が出揃ったとき」に走るPDCAループ(入力: 自分の投稿+数値)
- 出力先も別: ③→ `knowledge/reference_posts.md`、④→ `knowledge/performance_insights.md`

## ワークフロー

```
ユーザー ⇄ ①統括(指示・ナレッジ・報告のすべてがここを通る)
              ├─ ②リサーチャー ──生データ──→ ③競合アナリスト ──→ reference_posts.md
              ├─ ④自アカアナリスト(閲覧数データ) ──→ performance_insights.md
              └─ ⑤ライター ⇄ ⑥レビュアー(差し戻しは2回まで、超えたら統括判断)
                      ↑
        knowledge/ 全ファイルを参照(特に performance_insights.md の勝ち/負けパターン)
```

### 定常運用サイクル(推奨)

1. **収集**: ②が競合投稿を収集 → ③が分析 → 参考ポスト集へ昇格
2. **作成**: ⑤が投稿案を複数作成 → ⑥がチェック → 統括がユーザーに提案
3. **振り返り**: 投稿の閲覧数が出たら ④が要因分析 → 勝ち/負けパターン更新
4. 次の作成時、⑤は更新されたパターンを反映する(ここでループが閉じる)

## フォルダ構成

```
threads_post_system/
├── CLAUDE.md                      ← このファイル(全体方針・組織図)
├── prompts/                       ← 各ロールの指示書(1ロール=1ファイル)
│   ├── director.md
│   ├── researcher.md
│   ├── trend_analyst.md
│   ├── performance_analyst.md
│   ├── writer.md
│   └── reviewer.md
├── knowledge/                     ← チーム共有ナレッジ
│   ├── account_info.md            # 自アカ情報(コンセプト・口調・USP)
│   ├── teachings.md               # 教材から抽出した指示書
│   ├── reference_posts.md         # 参考ポスト集(③の分析から昇格)
│   ├── performance_insights.md    # 自アカ勝ち/負けパターン(④が更新)
│   ├── threads_principles.md      # Threads特有の作法
│   └── ng_patterns.md             # NGパターン・コンプライアンス
├── data/                          ← 分析用の生データ
│   ├── competitor_posts/          # ②が収集した競合投稿
│   └── my_posts/                  # 自アカ投稿+閲覧数の記録
└── logs/                          ← 生成物・作業ログ(output_[日付]_[内容].md)
```

## 共通ルール(全ロール)

1. 出力は必ず `knowledge/account_info.md` の口調・温度に従う
2. `knowledge/ng_patterns.md` のNG項目に該当しないか確認する
3. 参考ポストの構造は参考にしつつ **コピーはしない**
4. 同じ構文・冒頭フックを連続で使わない
5. 生成物・分析結果は `logs/output_[日付]_[内容].md` に保存する
6. ナレッジファイルへの追記は **統括経由でユーザー確認を取ってから** 行う

## X用システムとの関係

本システムは `x_post_system/` から独立して運用する。
ただしプラットフォーム非依存の原則(フック設計・教育の考え方など)は
`x_post_system/knowledge_common/` を副読本として参照してよい。
自アカが占いスピ系のため、`x_post_system/knowledge_spiritual/teachings.md`
(占いスピ系教材)も③⑤⑥は副読本として参照すること。
Threads特有の作法は必ず `knowledge/threads_principles.md` を優先し、
X前提の記述(タグ運用・RT文化など)はそのまま持ち込まない。

## 立ち上げロードマップ(統括が進捗を管理し、完了したらチェックを付ける)

### フェーズ0: アカウント開設前(ナレッジ先行)

自アカはまだ存在しない。先にジャンルの土地勘を作り、その知見でアカウントを設計する。

- [ ] 0-1. 参入ジャンル・方向性のヒアリング(仮でよいので `account_info.md` に記録)
- [ ] 0-2. 教材・ノウハウの取り込み(`knowledge/teachings.md`)※なければスキップ可
- [ ] 0-3. ウォッチする競合の選定(3〜5アカウント。選定基準は `prompts/researcher.md`)
- [ ] 0-4. 競合投稿の収集(②)と横断分析(③)→ ジャンルの勝ちパターン初期仮説を作る
- [ ] 0-5. 分析結果を踏まえてアカウントコンセプト・口調を設計(`account_info.md` 本記入)
- [ ] 0-6. アカウント開設・初期投稿案の作成(⑤→⑥→提案)

### フェーズ1: 運用開始後

- [ ] 1-1. 投稿+閲覧数を `data/my_posts/` に記録開始
- [ ] 1-2. ④自アカアナリストによる初回分析(閲覧数が出揃った投稿5本以上が目安)
- 以降: 作成→投稿→閲覧数共有→振り返り分析、の定常サイクル
